把URI（统一资源定位符）硬编码（hardcode）会增加测试程序的难度：表示路径的字段并不总是能跨操作系统的，给定的绝对路径在具体测试环境下可能并不存在，给定的Internet URL在程序测试时可能无效，生产环境的文件系统可能与与开发环境不同……等等。由于以上的原因，URI绝不应该硬编码，而是应该换用为可定制的参数。

而且，即使某个URI中的组成部分是动态获取（obtained dynamically）的，如果路径分隔符被硬编码，该条URI的通用性（portability）还是会受到限制。

当URI或路径分隔符被硬编码时，此规则会引发一个问题。

例外

ASP.NET的虚拟路径用作以下方法或函数的参数时不必考虑此规则：

  • 这些方法：System.Web.HttpServerUtilityBase.MapPath()，System.Web.HttpRequestBase.MapPath()， System.Web.HttpResponseBase.ApplyAppPathModifier()， System.Web.Mvc.UrlHelper.Content()
  • System.Web.VirtualPathUtility 的所有方法
  • 这些对象的构造函数：Microsoft.AspNetCore.Mvc.VirtualFileResult，Microsoft.AspNetCore.Routing.VirtualPathData

请参阅

  • CERT, MSC03-J. - Never hard code sensitive information
